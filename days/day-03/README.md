## Task

Fix a broken `requirements.in` for the fraud-detection project, then compile it into a pinned lockfile with `uv`.

**Acceptance criteria:**

- `/root/code/fraud-detection/requirements.in` lists exactly four top-level packages: `scikit-learn`, `mlflow`, `pandas`, `numpy`.
- Each carries a version constraint that `uv` can actually resolve against PyPI.
- `uv pip compile requirements.in -o requirements.txt` succeeds.
- The resulting `requirements.txt` pins every top-level package with `==` and includes the transitive deps `uv` resolved.

## Why this matters

`pip freeze` (Day 1) records everything installed â€” no distinction between "what I asked for" and "what was pulled in." The `requirements.in` â†’ `requirements.txt` split fixes that:

- **`requirements.in`** â€” the *spec*. Hand-edited. Direct dependencies, loose constraints. Source of truth for intent.
- **`requirements.txt`** â€” the *lockfile*. Machine-generated by `uv pip compile`. Every package pinned with `==`. Source of truth for reproducibility.

`uv` is ~10â€“100Ă— faster than `pip-tools`, same file formats. See [`notes/python-packaging.md` â€” Pinning, ranges, lockfiles](../../notes/python-packaging.md) for the conceptual treatment.

## How to diagnose

The lab's broken `requirements.in`:

```
sklearn
mlflow>=100.0
numpy
```

| Line | Problem | Fix |
|---|---|---|
| `sklearn` | Wrong PyPI name (`sklearn` is a deprecated stub that errors on install); no constraint | `scikit-learn>=1.4,<2` |
| `mlflow>=100.0` | Unsatisfiable (mlflow is ~2.x) | `mlflow>=2.10,<3` |
| `numpy` | No version constraint | `numpy>=1.26,<3` |
| *(missing)* | `pandas` not listed | add `pandas>=2.0,<3` |

The `sklearn` vs `scikit-learn` trap: the *import name* is `sklearn`, the *PyPI package* is `scikit-learn`. The `sklearn` stub on PyPI now raises on install to stop the confusion from spreading.

A reference [`requirements.in`](requirements.in) is in this directory.

## How to run

```bash
cd /root/code/fraud-detection
# fix requirements.in (see diagnosis table)
uv pip compile requirements.in -o requirements.txt

# verify
grep -E '^(scikit-learn|mlflow|pandas|numpy)==' requirements.txt
wc -l requirements.txt   # >> 4 (transitive deps included)
```

[`compile.sh`](compile.sh) wraps this.

## Key gotchas

- **`sklearn` vs `scikit-learn`.** Import name â‰  PyPI name. The stub raises on install.
- **Don't hand-edit `requirements.txt`.** It's generated output. Edit `requirements.in` and recompile.
- **Lockfiles are platform-specific by default** â€” compile on the platform you deploy on, or use `--universal`.
- **Hashes for production.** `uv pip compile --generate-hashes` adds SHAs; pip then refuses any wheel that doesn't hash-match.
- **`uv pip compile` vs `uv lock`.** Two different worlds â€” `uv pip compile` is pip-tools-style (this task). `uv lock` is the modern `pyproject.toml` + `uv.lock` flow used from Day 7 onwards.
