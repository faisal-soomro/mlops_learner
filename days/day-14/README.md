# Day 14 — Create a DVC Pipeline for Data Processing

**TL;DR:** Fix three plants in `dvc.yaml` — wrong script name in `cmd`, wrong output filename in `outs`, missing dep wiring between stages — then `dvc repro`. Don't touch the Python files.

## Task

A draft `dvc.yaml` exists with two stages (`process_data`, `split_data`) but `dvc repro` doesn't complete. Fix the YAML so the pipeline runs end to end and `dvc status` reports nothing stale.

### Acceptance criteria

- `process_data` stage:
  - `cmd: python src/data/process_data.py`
  - `deps: data/raw/transactions.csv, src/data/process_data.py`
  - `outs: data/processed/clean_transactions.csv`
- `split_data` stage:
  - `cmd: python src/data/split_data.py`
  - `deps: data/processed/clean_transactions.csv, src/data/split_data.py`
  - `outs: data/processed/train.csv, data/processed/test.csv`
- `dvc repro` runs both stages end to end with no errors
- `dvc status` reports no stale stages

## Starting state (the plants)

```yaml
stages:
  process_data:
    cmd: python src/data/process.py           # ← wrong script filename
    deps:
      - data/raw/transactions.csv
      - src/data/process_data.py
    outs:
      - data/processed/clean.csv              # ← wrong output filename
                                                # (the script writes clean_transactions.csv)
  split_data:
    cmd: python src/data/split_data.py
    deps:
      - src/data/split_data.py                # ← missing the upstream output
                                                # (clean_transactions.csv) as a dep
    outs:
      - data/processed/train.csv
      - data/processed/test.csv
```

The scripts (we read them but don't edit them) confirm the real filenames:
- `process_data.py` writes `data/processed/clean_transactions.csv`
- `split_data.py` reads `data/processed/clean_transactions.csv`, writes `train.csv` + `test.csv`

So the truth is in the scripts; the YAML is misaligned with reality on three counts.

## Diagnosis table

| Symptom (running `dvc repro`) | Root cause | Fix |
|---|---|---|
| `python: can't open file 'src/data/process.py'` | Typo in `cmd` | Change to `src/data/process_data.py` |
| Stage runs, but `outs` doesn't match what the script wrote → DVC error `missing data: data/processed/clean.csv` | Output filename in YAML doesn't match what the script produces | Change `outs` to `data/processed/clean_transactions.csv` |
| `dvc status` reports `split_data` as not depending on the upstream output | Missing dep wiring | Add `data/processed/clean_transactions.csv` to `split_data.deps` |

The third one is the conceptual heart of the lab: **`deps` is how DVC builds the DAG**. Without that line, DVC thinks the two stages are independent and won't re-run `split_data` when `clean_transactions.csv` changes. A pipeline that produces correct outputs *by luck of execution order* but doesn't declare the dep is broken — same as a Makefile target without the right prerequisites.

## How to run

```bash
cd /root/code/fraud-detection

# Edit dvc.yaml — three fixes:
#   1. process_data.cmd: process.py → process_data.py
#   2. process_data.outs: clean.csv → clean_transactions.csv
#   3. split_data.deps: add data/processed/clean_transactions.csv

# (use vim/nano/whatever — there's no `dvc stage modify` for editing in-place)

# Run the pipeline
dvc repro

# Check nothing is stale
dvc status

# Inspect the DAG
dvc dag
```

## Gotchas

- **`dvc repro` is idempotent.** If nothing changed, it skips both stages with "Stage 'X' didn't change, skipping". To force re-run: `dvc repro --force`.
- **`outs` filenames must match what the script writes exactly.** DVC doesn't introspect the script; it just checks the named output file exists after the command runs. Mismatch → error.
- **`deps` filenames must match what the script reads, AND must be either real files or other stages' `outs`.** A `dep` that points at a non-existent path that no upstream stage produces will fail at parse time.
- **`dvc repro` auto-tracks the outputs.** You don't `dvc add` them — outputs declared in a stage are managed by DVC automatically. They show up in `dvc.lock` (the pipeline's MD5 manifest).
- **Hand-editing `dvc.yaml` is the normal workflow.** Unlike `.dvc/config`, where `dvc remote modify` is preferred, there's no equivalent `dvc stage modify`. Use an editor. (There is `dvc stage add` for creating new stages from scratch — useful in fresh projects, not for editing existing ones.)
- **`dvc.yaml` is YAML, not INI.** Indentation matters. Two spaces, consistent throughout.
- **`dvc.lock` is auto-generated by `dvc repro`.** It records the actual hashes of each stage's deps + outs after running. Commit `dvc.lock` alongside `dvc.yaml` — that's how the pipeline becomes reproducible across machines.

## Why this matters

`dvc add` (Days 11-13) versions individual files. `dvc.yaml` versions **the relationship between files** — which script transforms which input into which output. The pipeline is a Makefile written in YAML, with content-hashing on every node.

The payoff: when `data/raw/transactions.csv` changes, `dvc repro` re-runs `process_data` (because its dep hash changed), then re-runs `split_data` (because *its* dep — `clean_transactions.csv` — was just regenerated by `process_data`). Stages whose deps are unchanged are skipped. This is the entire point of pipelines: re-run what's stale, skip what isn't.

Without the dep wiring (the third plant), DVC can still execute both stages if you ask, but it can't tell that `split_data` is *stale* when `transactions.csv` changes. So `dvc status` won't flag it, `dvc repro` will skip it, and you'll silently train on stale data.

## Use case

You're iterating on data cleaning logic. You tweak `process_data.py`, re-run `dvc repro`. DVC sees the script's hash changed (it's a `dep`), re-runs `process_data`, then automatically re-runs `split_data` because its input changed. Two minutes later you have fresh `train.csv` / `test.csv`. Without DVC: you'd have to remember which downstream scripts depend on what and re-run them by hand. With 20+ stages and a team of three, that's the kind of mistake that puts wrong data into a model that goes to prod.
