# ML project layout — the standard skeleton

Cross-cutting notes on the conventional directory shape for an ML project. A consistent layout is the cheapest piece of MLOps infrastructure you can buy: it pays off every time someone new opens the repo, and every later tool (DVC, MLflow, Docker, CI) expects to find things in predictable places.

## The skeleton

```
my-project/
├── data/
│   ├── raw/             # immutable input — never edit, never overwrite
│   └── processed/       # derived output — regenerable from raw/ + code
├── models/              # trained model artifacts (gitignored; tracked by MLflow/DVC)
├── notebooks/           # exploratory work; throwaway
├── src/                 # the library — importable, testable, reviewed
│   ├── data/            # loading, cleaning
│   ├── features/        # feature engineering
│   ├── models/          # training, evaluation
│   └── utils/           # cross-cutting helpers
├── tests/               # pytest tests for src/
├── configs/             # hyperparameters, data paths, run configs (YAML/JSON)
├── requirements.txt     # dependency spec (or pyproject.toml)
└── README.md
```

Every subdirectory under `src/` contains an `__init__.py` (even if empty).

## What each directory is for

### `data/raw/` vs `data/processed/`

The most load-bearing split in the layout.

- **`data/raw/`** is *immutable input*. The file you downloaded from S3, the CSV the analytics team handed you, the parquet from Kafka. You never edit it in place; you never overwrite it. If you need a fix, you regenerate the pipeline upstream and re-ingest.
- **`data/processed/`** is *derived output* — generated by `src/data/` and `src/features/` from `raw/` + code. Regenerable.

The split is the precondition for reproducibility: if `processed/` is lost, you can rebuild it; if `raw/` is lost, you have a real problem. When DVC arrives (Days 10–19), it tracks both — and assumes this split.

### `src/` vs `notebooks/`

- **`src/`** is the library. Importable, testable, version-controlled, code-reviewed.
- **`notebooks/`** is exploration. Messy, throwaway, useful for plots and one-off analysis.

The boundary moves over time: logic that matters migrates from `notebooks/` to `src/` once it stops being exploratory. Notebooks that ship to production are a smell; the value of the notebook was the exploration, not the artifact.

### Slicing `src/` by lifecycle stage

`src/data/` (loading, cleaning), `src/features/` (feature engineering), `src/models/` (training, evaluation), `src/utils/` (cross-cutting helpers).

Not the only way to slice it, but it matches the stages of an ML pipeline. People find things by intent ("where do features live?") rather than by guessing at module names.

### `tests/` and `configs/`

- **`tests/`** mirrors `src/`. `pytest tests/` runs the whole suite. Without a `tests/` directory at all, the first reviewer to ask "where are the tests?" will not be impressed.
- **`configs/`** separates *what to run* from *how to run it*. Hyperparameters, data paths, model names — in YAML/JSON, not hardcoded. Day 32 will lean on this when training is wired to a YAML config.

### `models/`

Trained model artifacts land here when you're not yet wired up to MLflow or DVC. Once those tools are in (Days 10+ for DVC, Days 20+ for MLflow), `models/` becomes a local staging area and the real model store lives elsewhere.

The directory itself should exist (so scripts can write to it); the *contents* should be gitignored.

## `__init__.py` — packages vs namespace packages

`__init__.py` marks a directory as a Python *package*. With it, `from src.models.train import train_model` works unambiguously.

Without it, Python 3.3+ treats the directory as a *namespace package* (PEP 420). Namespace packages mostly work but break in subtle ways:

- Some test runners and packaging tools don't discover them correctly.
- Editable installs (`pip install -e .`) can pick up the wrong path.
- Tooling that walks `__all__` or generates docs may miss modules.

The safer default is explicit `__init__.py` everywhere. Empty file is fine — its presence is what matters. Every linter, test runner, and packaging tool understands this; not all understand namespace packages.

## `src/` layout vs flat layout

Two ways to lay out Python projects:

- **Flat layout** — package code sits at the repo root (`my_project/__init__.py` next to `tests/`).
- **`src/` layout** — package code is one level down (`src/my_project/__init__.py`).

`src/` layout has two concrete benefits:

- **Forces installation before import.** Without `src/`, `python -c "import my_project"` from the repo root works even if you forgot to `pip install -e .`. With `src/`, it doesn't — which means tests run against the *installed* package the same way users will. No more "works on my machine because my CWD happens to put the source on `sys.path`."
- **Prevents accidental imports from a half-installed package.** Same mechanism — if installation is broken, imports fail loudly instead of silently shadowing.

Flat layout is fine for small scripts. For anything that will be packaged and distributed (Day 7 onwards), `src/` is the standard. See `notes/python-packaging.md` for the packaging side of this.

## `requirements.txt` here is a spec, not a lockfile

A bare `requirements.txt` listing unpinned names (`scikit-learn`, `pandas`, ...) is a *spec* — same role as `requirements.in`. The *lockfile* is a separately compiled, fully-pinned file produced by `uv pip compile`, `pip-tools`, or `poetry lock`.

Both have a place in a project:

- The spec is what humans edit.
- The lockfile is what CI installs from.

For a layout-only exercise, an unpinned spec is enough. In a real project, you'd compile this to a lockfile and commit both. See `notes/python-packaging.md` "Pinning, ranges, lockfiles" for the full discussion.

## Don't put data in git

`data/raw/` and `data/processed/` should be gitignored. Git's content-addressed storage handles small text files brilliantly and large binaries terribly — a 2 GB CSV ends up in git history forever, slowing every clone, and you still can't `diff` it usefully.

The MLOps answer is DVC (Days 10–19), which stores blobs in S3/GCS/Azure and tracks pointer files in git. Until then: `.gitignore` the data, document where it comes from in the README, and accept that "the repo is not self-contained" — it's a code repo, not a data archive.
